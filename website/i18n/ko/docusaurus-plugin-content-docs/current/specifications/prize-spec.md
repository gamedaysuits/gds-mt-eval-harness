# 상금 명세

> **목적.** 이 문서는 MT Eval Arena의 상금 풀 구조, 임계 조건, 청구 절차, 규칙을 정의해요. "기계 번역이 가능하다"는 것이 측정 가능한 관점에서 정확히 무엇을 의미하는지, 그리고 어떤 조건에서 상금이 지급되는지를 명시해요. 이 문서는 메트릭 정의에 대해 SCORING_SPEC을, 평가 프로토콜에 대해 BENCHMARK_SPEC을 참조하며, 해당 내용을 중복하지는 않아요.
>
> **상태:** LIVE. Founder's Prize(§2.1)는 자금이 확보되어 활성화되어 있어요.
>
> 마지막 업데이트: 2026-06-04

---

## 1. 철학

### 1.1 상금은 참가가 아니라 돌파를 보상해요

상금은 어떤 방법이 정의된 역량 임계값에 명확히 도달했을 때에만 지급돼요. 참가상, 준우승상, 위로금은 없어요. 아무도 기준을 넘지 못하면 아무도 받지 못해요. 이는 의도된 설계예요 — 후원자가 실제로 작동하는 결과에 대해서만 비용을 지불한다는 의미예요.

### 1.2 커뮤니티 검증은 타협할 수 없어요

자동화된 메트릭은 대리 지표예요(SCORING_SPEC §1.1). 어떤 방법이 chrF++와 FST 수용률에서 좋은 점수를 받으면서도, 어떤 화자도 받아들이지 않을 출력물을 만들어낼 수 있어요. **모든 상금 청구는 커뮤니티 검증을 필요로 해요** — 이중 언어 화자가 출력물이 사용 가능한지 확인해야 해요. 이것이 인간 검증 관문(BENCHMARK_SPEC §7)이에요.

### 1.3 소유권 이전은 거래의 일부예요

상금을 청구하는 방법은 소유권 이전 조항(BENCHMARK_SPEC §8.3)의 적용을 받아요. 개발자는 저작자 표시 및 발행 권리를 유지해요. 거버넌스 조직은 해당 언어에 대해 그 방법을 사용, 수정, 배포, 수익화할 권리를 얻어요. 이는 벌칙이 아니라 — 핵심이에요. 상금은 언어 커뮤니티에 속하는 기술의 창출에 자금을 지원해요.

### 1.4 부정행위 방지

상금 임계값은 **골드 스탠다드 평가**(비공개 테스트 세트, 거버넌스 조직이 샌드박스에서 실행)를 기준으로 정의돼요. 개발자는 테스트 데이터를 절대 보지 못해요. 이는 명예에 의존하는 정책이 아니라 — 아키텍처적으로 강제돼요. BENCHMARK_SPEC §8.2를 참조하세요.

### 1.5 코퍼스 라이선싱: 비상업적 코퍼스는 상금 레인 밖에 머물러요

방법 개발 중에 사용되는 일부 코퍼스는 비상업적 라이선스를 가지고 있어요 — 예를 들어, EdTeKLA Cree Language Textbook 코퍼스는 **CC BY-NC-SA 4.0**이에요. 이러한 코퍼스는 **연구/개발 레인 전용**이에요:

1. **상금 골드 스탠다드 코퍼스는 NC 라이선스 코퍼스 콘텐츠를 포함해서는 안 돼요.** 골드 스탠다드 테스트 세그먼트는 커뮤니티가 의뢰한 원본 자료예요(Corpus Partnership Strategy 참조) — 상금을 위해 인간이 작성하며, 처음부터 평가 및 상업적 배포에 대한 권리가 정리되어 있어요.
2. **상금을 청구하는 방법은 NC 라이선스 코퍼스 콘텐츠를 포함해서는 안 돼요**(예: 코칭 데이터, 임베디드 예시, 룩업 테이블로). 이전된 방법은 거버넌스 조직에 의한 상업적 배포를 의도하고 있어요(BENCHMARK_SPEC §8.3, Method Submission Agreement §6); 그 안에 포함된 NC 라이선스 콘텐츠는 그 배포를 오염시킬 거예요.
3. **개발자는 NC 라이선스 코퍼스를 개발 및 자체 평가에 자유롭게 사용할 수 있어요** — 그것이 바로 개발 레인의 목적이에요. 이 제한은 제출되는 것과 배포되는 것에 적용되며, 개발자가 학습하는 방식에는 적용되지 않아요.

### 1.6 의존성 클래스가 상금 자격을 결정해요

모든 상금 평가는 샌드박스에서 이루어지며(§1.4), 상금을 수상한 방법은 거버넌스 조직으로 이전돼요(§1.3). 두 사실 모두 동일한 제약을 부과해요: **방법이 의존하는 모든 것은 개발자가 샌드박스에 넣고 커뮤니티에 양도할 권리가 있는 것이어야 해요.** 모든 제출물은 의존성 클래스를 선언해요 — 이는 [Method Interface spec](/docs/specifications/methods#method-validity-and-dependency-classes)에 정의되어 있고, 허용 조건은 Method Submission Agreement §2.6에 있으며 — 자격은 클래스에 따라 결정돼요:

| 의존성 클래스 | 상금 자격 여부? | 조건 |
|------------------|----------------|------------|
| **S** — 자체 완결형 | ✅ 예 | §2의 임계 조건 외에 없음 |
| **O** — 개방형 외부(예: 제출 시 미러링된 AGPL FST) | ✅ 예 | 아티팩트가 고정되어 제출물에 벤더링됨; 라이선스가 커뮤니티 이전을 허용함; 카피레프트 조건이 보존됨(커뮤니티는 라이선스가 모두에게 부여하는 것과 동일한 권리를 받음) |
| **A1** — 대체 가능한 LLM 추론 | ⚠️ 조건부 | 모델이 선언, 고정, 대체 가능함(커뮤니티 호스팅 오픈 웨이트 모델에 대해 실행되어야 함); 평가는 샌드박스 LLM 게이트웨이를 통해 라우팅됨(🔲 계획됨 — A1 방법은 게이트웨이가 운영되기 전까지 골드 스탠다드 점수를 생성할 수 없음); 이전 시 모델이 아니라 전체 레시피(프롬프트, 코칭, 코드)가 전달됨 |
| **A2** — 대체 불가능한 외부 데이터/서비스 API | ❌ 아직 아님 | 권리 보유자가 샌드박스 포함 및 이전 권한을 부여할 때까지 자격 없음. 가시적인 "external dependency" 플래그와 함께 오픈 리더보드에서는 허용됨 |
| **X** — 권리 없는 번들 콘텐츠 | ❌ 절대 아님 | 모든 레인에서 허용 불가 |

방법의 클래스는 선언된 의존성 중 가장 제한적인 클래스예요. 클래스를 불문하고 선언되지 않은 의존성은 실격 사유예요(§5).

## 2. 활성 상금 풀

### 2.1 The Founder's Prize — EN→Plains Cree (nêhiyawêwin)

| 필드 | 값 |
|-------|-------|
| **상금 풀** | **$10,000 CAD** |
| **언어 쌍** | English → Plains Cree (EN→CRK) |
| **자금 지원** | Champollion 프로젝트 창립자 |
| **상태** | **ACTIVE** — 제출 접수 중 |
| **개시** | 골드 스탠다드 코퍼스 + 거버넌스 조직이 준비될 때 |
| **만료** | 만료 없음. 청구되거나 명시적으로 철회될 때까지 상금은 활성 상태로 유지돼요. |

#### 임계 조건

어떤 방법은 다음 조건을 **모두** 동시에 충족함으로써 Founder's Prize를 청구해요:

| # | 조건 | 메트릭 | 임계값 | 근거 |
|---|-----------|--------|-----------|-----------|
| 1 | **복합 점수** | `composite` (SCORING_SPEC §4) | **≥ 0.80** | Deployable(0.70)과 Fluent(0.85) 사이. 형태론적 유효성뿐만 아니라 모든 메트릭 차원에 걸쳐 높은 품질을 요구해요. |
| 2 | **FST 수용률** | `fst_acceptance_rate` (SCORING_SPEC §2.2) | **≥ 0.99 (99%+)** | 사실상 모든 출력 단어가 GiellaLT FST가 인식하는 형태론적으로 유효한 형태여야 해요. 1% 허용 범위는 FST가 정당하게 다루지 못할 수 있는 엣지 케이스(고유명사, 신조어, 차용어)를 감안한 거예요. 이는 다종합어 MT의 결정적인 품질 관문이에요 — FST가 단어의 1% 이상을 거부하면, 그 방법은 해당 언어에 존재하지 않는 형태를 생성하고 있는 거예요. 이 상금의 전체 핵심은 무언가를 망가뜨리지 않는 시스템을 사들이는 데 있어요. |
| 3 | **chrF++** | `chrf_plus_plus` (SCORING_SPEC §2.1) | **≥ 55.0** | 문자 n-gram 중첩이 0–100 척도에서 55를 초과해야 해요. 형태론적 유효성뿐만 아니라 참조 번역과의 표면 수준 유사성을 보장해요. |
| 4 | **커뮤니티 검증** | 인간 검토(BENCHMARK_SPEC §7) | **≥ 70% "acceptable" 또는 "excellent"** | 출력물의 층화 표본(난이도 등급 2–5에 걸쳐 ≥30개 항목)을 ≥2명의 이중 언어 CRK 화자가 검토해요. 검토된 항목의 최소 70%가 "acceptable" 또는 "excellent" 평가를 받아야 해요. |
| 5 | **골드 스탠다드 평가** | 샌드박스 실행(BENCHMARK_SPEC §8.2) | **필수** | 모든 자동화된 메트릭은 거버넌스 조직이 샌드박스 환경에서 실행하는 `gold_standard` 코퍼스 세그먼트를 기준으로 계산되어야 해요. 개발 세트 점수는 인정되지 않아요. |
| 6 | **재현성** | 핑거프린트 일치(BENCHMARK_SPEC §3.8) | **±2%** | 거버넌스 조직이 방법을 다시 실행해 제출된 런 카드의 ±2% 이내 점수를 달성할 수 있어야 해요. |

> **왜 99+% FST인가요?** 다종합어 기계 번역의 핵심 문제는 환각이에요 — LLM은 대상 언어처럼 *보이지만* 형태론적으로 유효하지 않은 문자열을 생성해요. 95% 유효 출력을 생성하는 방법도 여전히 5%의 조작된 단어를 가지고 있어요 — 어떤 프로덕션 용도에든 받아들일 수 없는 노이즈예요. 99%+ 임계값은 드문 엣지 케이스(FST가 모르는 고유명사, 정당한 신조어)를 허용하면서도 거의 제로에 가까운 환각을 요구해요. 어떤 방법이 99%+ FST 수용률을 달성할 수 없다면, 그것은 문제를 해결하지 못한 거예요.
>
> **왜 0.80 복합 점수인가요?** 이는 Deployable(0.70)과 Fluent(0.85) 사이에 위치해요. 99%+ FST 수용률을 가진 0.80의 방법은 사실상 모든 단어가 실제 Cree 단어*이면서도* 표면, 구조, 의미 차원 전반에 걸쳐 전체 번역 품질이 높은 출력물을 생성해요. 커뮤니티 검증 관문(조건 #4)은 이것이 단순한 메트릭 게이밍이 아님을 보장해요 — 화자들이 출력물이 진정으로 사용 가능한지 확인해야 해요.

#### 이 임계값이 실제로 의미하는 것

복합 점수 ≥ 0.80, FST ≥ 0.99, chrF++ ≥ 55에서, 이중 언어 화자는 일반적으로 다음을 보게 돼요:

- **사실상 모든** 출력 단어가 실제 Cree 단어임(FST가 99%+를 검증 — 거의 제로에 가까운 환각 형태)
- 주요 문법 범주(인칭, 수, 시제)가 대부분의 항목에서 올바름
- 어순이 대체로 자연스러움
- 의미가 신뢰성 있게 보존됨
- 남은 오류는 조작된 단어가 아니라 실제 언어 오류(잘못된 굴절, 부정확한 obviation, 유생성 불일치)임
- 유창한 화자가 출력물을 고품질 초안으로 사용하고, 처음부터 번역하는 것보다 훨씬 빠르게 수정할 수 있음

이는 **언어를 망가뜨리지 않는** 시스템이에요. 완벽하지는 않을 수 있지만, 그것이 생성하는 모든 단어는 실제 단어예요. 그것이 다종합어의 존중받는 기계 번역을 위한 최소 기준이에요.

## 3. 상금 청구 절차

### 3.1 제출

1. 개발자가 완전하고 실행 가능한 방법을 거버넌스 조직에 제출해요:
   - 모든 소스 코드
   - 모든 의존성(코칭 데이터, 사전, FST 설정, 프롬프트)
   - 설치 및 실행 지침
   - 방법의 접근 방식을 설명하는 README
   - 대략적인 점수를 보여주는 개발 세트 런 카드(사전 심사용)

2. 개발자가 다음을 포함한 참가 약관에 서명해요:
   - 소유권 이전 조항(BENCHMARK_SPEC §8.3)
   - 평가 데이터로 학습하지 않았다는 선언
   - 재현성 약속

### 3.2 평가

1. 거버넌스 조직이 `gold_standard` 코퍼스를 기준으로 샌드박스 하니스에서 방법을 설치하고 실행해요
2. 자동화된 메트릭이 계산돼요(복합, FST, chrF++ 등)
3. 자동화된 임계값이 충족되면(조건 1–3), 거버넌스 조직이 커뮤니티 검토로 진행해요
4. 자동화된 임계값이 충족되지 않으면, 개발자는 점수와 피드백을 받아요. 커뮤니티 검토는 트리거되지 않아요.

### 3.3 커뮤니티 검토

1. 출력물의 층화 표본(≥30개 항목, 난이도 등급 2–5 포함)이 이중 언어 화자에게 제시돼요
2. 최소 2명의 독립적인 검토자가 각 항목을 평가해요
3. 평가 척도: **reject** / **gist** / **acceptable** / **excellent**
4. 항목의 ≥70%가 두 검토자 모두로부터 "acceptable" 또는 "excellent"를 받으면, 커뮤니티 검증을 통과해요

### 3.4 지급

1. 6개 조건이 모두 충족돼요
2. 거버넌스 조직이 결과를 확인해요
3. 확인 후 30일 이내에 상금이 지급돼요
4. 방법 소유권이 BENCHMARK_SPEC §8.3에 따라 이전돼요
5. 결과가 "Community Validated" 검증 등급과 함께 리더보드에 게시돼요

### 3.5 다중 제출

- 동일한 개발자/팀이 여러 번 제출할 수 있어요
- 각 제출물은 독립적으로 평가돼요
- 방법이 개선되어 재제출되면, 가장 최근의 런 카드만 인정돼요
- 상금은 모든 임계값을 통과한 **첫 번째** 방법에게 수여돼요 — 분할되지 않아요

### 3.6 팀 제출

- 팀과 Elder-youth 쌍이 자격이 있어요
- 팀 내 상금 분배는 팀의 책임이에요
- 모든 팀원이 참가 약관에 서명해야 해요
- 리더보드의 저작자 표시에는 모든 팀원이 나열돼요

## 4. 향후 상금 풀

Founder's Prize는 씨앗이에요. 추가 상금 풀은 후원자가 자금을 지원해요. 각 새로운 상금 풀은 §2의 새로운 하위 섹션으로 문서화되며, 다음과 같은 자체 요소를 가져요:

- 상금 금액 및 통화
- 언어 쌍
- 후원자 저작자 표시
- 임계 조건(Founder's Prize와 다를 수 있음)
- 만료일(있는 경우)
- 모든 특별 조건

### 4.1 후원자 상금 템플릿

후원자는 어떤 금액으로든 상금 풀에 자금을 지원해요. 제안 등급:

| 등급 | 금액 | 제안 임계값 |
|------|--------|---------------------|
| **Seed** | $5,000–$15,000 | Deployable(복합 점수 ≥ 0.70) + 커뮤니티 검증 |
| **Breakthrough** | $25,000–$50,000 | Fluent(복합 점수 ≥ 0.85) + 커뮤니티 검증 |
| **Grand Prize** | $100,000+ | Fluent + 다중 레지스터 커버리지 + 배포 통합 |

후원자는 또한 다음에 자금을 지원할 수 있어요:
- **개선 현상금** — 현재 최고치 대비 chrF++가 5점 개선될 때마다 지급되는 고정 금액
- **레지스터 상** — 특정 레지스터(격식체, 의례체, 교육체)에 대한 별도의 상
- **속도 상** — 최고의 비용 조정 점수(SCORING_SPEC §6.3)

### 4.2 상금 풀 에스크로

모든 상금 자금은 임계 조건이 충족될 때까지 에스크로(프로젝트 또는 지정된 수탁자가 관리)로 보관돼요. 상금이 청구되지 않은 채 만료되면, 자금은 후원자에게 반환되거나 후원자의 재량에 따라 새로운 상금 풀로 전환돼요.

## 5. 실격

다음의 경우 제출물이 실격돼요:

1. **평가 데이터로 학습.** 방법이 `gold_standard` 또는 `held_out` 코퍼스 항목에 노출됨. (샌드박스 실행으로 아키텍처적으로 방지되지만 — 오염의 증거가 발견되면 결과가 무효화돼요.)
2. **재현 불가.** 거버넌스 조직이 ±2% 이내로 점수를 재현할 수 없음.
3. **선언되지 않았거나 자격 없는 의존성.** 방법이 의존성 매니페스트가 선언하는 것 이상의 외부 서비스에 대한 런타임 접근을 필요로 하거나, 그 유효 의존성 클래스가 A2 또는 X임(§1.6). 평가 게이트웨이를 통해 라우팅되는 선언된 Class A1 LLM 추론은 허용되지만; 그 외의 모든 런타임 네트워크 의존성 — 그리고 클래스를 불문하고 선언되지 않은 모든 의존성 — 은 실격 사유예요.
4. **참가 약관 미서명.** 모든 팀원이 소유권 이전에 동의해야 해요.
5. **게이밍 탐지.** 출력물이 번역 품질이 아니라 메트릭에 최적화됨(커뮤니티 검토 및/또는 BENCHMARK_SPEC §9.3에 따른 부정행위 방지 검사로 적발).

### 7.2 구현 요구사항

상금 풀이 활성화될 때:
1. 리더보드 UI는 활성 상금과 그 임계 조건을 표시해야 해요
2. 자동화된 임계값(조건 1–3)을 충족하는 런 카드는 커뮤니티 검토를 위해 플래그되어야 해요
3. 런 카드 스키마의 `quality_tier` 필드는 이미 등급("deployable", "fluent")을 캡처하고 있어요
4. 하니스에 대한 새로운 코드 변경은 필요하지 않아요 — 상금 명세는 기존 채점 위에 놓인 정책 계층이에요

---

*상금 구조는 소유권 이전 조건과 호환되어야 해요. 수상자는 상금을 청구할 수 있지만, 방법이 Deployable 등급에 도달하면 거버넌스 조직의 자산이 돼요. 이는 의도된 설계예요 — 상금은 언어 커뮤니티에 속하는 기술의 창출에 자금을 지원해요.*
